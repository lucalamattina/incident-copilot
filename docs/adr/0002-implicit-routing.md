# Implicit routing between Investigation and Retrieval modes

The agent is given all tools (the deploys/logs query tools and the playbook retrieval tool) and decides which to call based on the engineer's message. There is no separate classification step that labels a request as investigation-vs-retrieval before acting.

Implicit routing was chosen for flexibility: it is simpler, and it lets a single message legitimately blend both modes ("the db looks slow, what does the playbook say") without forcing an artificial binary label. The rejected alternative, explicit routing via an upfront classifier, would be more predictable and easier to evaluate, since each request would carry a known intent. We accept the cost: evaluation must handle the agent's tool-selection freedom (see the tiered eval) rather than asserting against a known intent label. Concretely, cleanly single-intent prompts are asserted exactly at Tier one, and blended prompts (the case that justifies implicit routing in the first place) are asserted at Tier two by a coverage property: the agent exercised both modes, order-agnostic, checkable programmatically from the trace. So the strongest argument for implicit routing is itself a covered evaluation case rather than a hole.
